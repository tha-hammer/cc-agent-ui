# Context for gwt-0002: fork.handler.slices-parent-messages

## Behavior
Given handleForkFromMessage is invoked with a valid messageId (string) and messageIndex (number) while parentSessionId is the current active session in useSessionStore, when handleForkFromMessage reads the parent session's message history, then it calls useSessionStore.getState().sessions.get(parentId).merged.slice(0, messageIndex + 1), producing a slice that includes all messages up to and including the forked message, without mutating the parent session store entry.

## Key Invariants to Model
- SliceUpToAndIncluding: slice length === messageIndex + 1; last element.id === messageId.
- ParentImmutable: the SessionSlot.merged array reference at parentId is NOT reassigned and its contents equal the pre-call contents.
- ReadOnlyAccess: handler only reads via useSessionStore.getState(); no set/update calls against the parent slot.

## Modeling Approach
- CONSTANTS: ParentMessages (sequence), ForkIdx.
- Variables: parentSlot.merged, handlerSliceResult.
- Actions: InvokeFork(messageId, ForkIdx).
- Invariants: SliceUpToAndIncluding, ParentImmutable.

## Anti-Patterns (DO NOT USE)
- Using `splice` instead of `slice` (mutates parent).
- Calling `useSessionStore.getState().updateSession(...)` or setters against parent slot.
- Using `useSessionStore.getState()` or `useSessionStore.getState().sessions.set(...)` — `useSessionStore` is NOT a Zustand store. It is a plain React hook with no static `.getState()` accessor and no exposed `.sessions` Map. Seed via the hook's real API (`appendRealtime`) inside `renderHook` + `act`.
- Asserting `slot.merged !== merged` — a new reference would indicate accidental mutation by the store.
- Slicing with `messageIndex` instead of `messageIndex + 1` (off-by-one, omits the forked message).
