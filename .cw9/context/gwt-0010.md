# Context for gwt-0010: fork.nav.navigates-on-session-created

## Behavior
Given the Claude SDK emits a `session_created` event over WebSocket after processing the fork command carrying a new `sessionId` distinct from the parent, when `useChatRealtimeHandlers` processes the `session_created` event at line 242, then it calls `onNavigateToSession(newSessionId)` routing the UI to `/session/{newSessionId}`. Handler is reused verbatim; no modifications.

## Key Invariants to Model
- NavigateAlways: on every `session_created` with a truthy `newSessionId`, `onNavigateToSession(newSessionId)` is invoked exactly once.
- DistinctFromParent: for forks, `newSessionId !== parentSessionId` (server emits only fresh id).
- NoParentReplaceOnFork: `onReplaceTemporarySession` only fires when `currentSessionId` is null or startsWith `'new-session-'` (fork originates from a real parent, so this branch is skipped).

## Anti-Patterns (DO NOT USE)
- Guarding `onNavigateToSession` behind the `!currentSessionId || startsWith('new-session-')` branch â€” navigation must fire unconditionally for forks (parent is a real UUID).
- Asserting on router side effects via `window.location` â€” the hook only calls the injected callback; navigation is the consumer's responsibility.
- Calling `result.current.handleMessage(msg)` â€” this method does NOT exist on the hook's return. `useChatRealtimeHandlers` is prop-driven: it reads `latestMessage` from its deps and reacts via `useEffect`. Drive events by `rerender({ latestMessage: newMsg })`.
- Using `fireEvent` against a rendered DOM â€” this hook has no DOM; use `renderHook` + `rerender` to flip `latestMessage`.
