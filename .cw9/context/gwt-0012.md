# Context for gwt-0012: fork.settings.localStorage-not-cleared-on-nav

## Behavior
Given the user navigates from parent session P to the forked session F via `onNavigateToSession`, when `onNavigateToSession` executes (the call site is `useChatRealtimeHandlers.ts:257`), then localStorage keys `'claude-settings'`, `'cursor-tools-settings'`, `'codex-settings'`, `'gemini-settings'` are not cleared, deleted, or modified — the new session's composer reads the same settings that were active during the parent session.

## Concrete Data Shapes
```ts
// localStorage keys surveyed in composer layer
const WATCHED_KEYS = [
  'claude-settings',         // { model, allowedTools, disallowedTools, skipPermissions, permissionMode }
  'cursor-tools-settings',
  'codex-settings',
  'gemini-settings',
];

// Nav callback contract (external):
type OnNavigateToSession = (sessionId: string) => void;
// Consumer typically: navigate(`/session/${sessionId}`)
```

## Key Invariants to Model
- LocalStoragePersistAcrossNav: ∀ k ∈ WATCHED_KEYS, `localStorage.getItem(k)` before nav === after nav.
- NoRemoveItemCalls: no `localStorage.removeItem(k)` / `clear()` call issued during or within the synchronous fork-nav sequence.
- SessionStorageScoped: `sessionStorage` writes are limited to `'pendingSessionId'` (see line 247); unrelated storage untouched.

## Modeling Approach
- CONSTANTS: WATCHED_KEYS.
- Variables: `store` (map key → value), `removedKeys`, `setKeys`.
- Actions: `Navigate(newId)` invokes callback; observer records all `removeItem`/`setItem`/`clear` calls.
- Invariants: LocalStoragePersistAcrossNav, NoRemoveItemCalls.

## Boundary Contracts

### BC-4: Browser storage boundary (localStorage settings + sessionStorage pendingSessionId)

- **Emitter**: useChatRealtimeHandlers session_created handler writes sessionStorage.pendingSessionId; useChatComposerState reads localStorage.<provider>-settings
- **Event shape**: `localStorage.getItem(<provider-key>) → JSON|null; sessionStorage.setItem('pendingSessionId', UUID)`
- **Handler**: Storage.prototype methods (browser-native)
- **Result shape**: string|null read; void write; persistence across nav
- **Resume guard**: gwt-0012 verifies localStorage watched-keys never removed/cleared/rewritten during nav; sessionStorage scoped to pendingSessionId only
- **Covered by**: gwt-0004 (read path) + gwt-0012 (write path with spies)

## Callers
- `onNavigateToSession` callback: supplied by the chat container (e.g., `ChatInterface.tsx`), typically wrapping `react-router` `navigate`.
  Reuse risk: high — any future consumer wiring that does `localStorage.removeItem('claude-settings')` inside the nav callback would silently regress this GWT. Keep storage writes out of the nav path.

## Anti-Patterns (DO NOT USE)
- "Reset settings on session change" side effects in the nav callback.
- `sessionStorage.clear()` inside the session_created branch (line 247 only sets `pendingSessionId`; `clear()` would also wipe unrelated keys).
- Calling `localStorage.setItem('claude-settings', …)` inside the fork dispatch to "materialize" inherited settings — the whole point is they're already inherited.
- Calling `result.current.handleMessage(msg)` — no such method exists on `useChatRealtimeHandlers`. It is prop-driven; flip `latestMessage` via `rerender`.
